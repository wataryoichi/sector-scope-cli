## セッション開始時の必須手順
- 作業開始時は、まず `docs/spec.md` を確認すること。
- `TODO.md` が存在する場合は必ず確認すること。
- `docs/handoff-current.md` が存在する場合は必ず確認すること。
- 仕様を確認する前に実装を開始しないこと。
- 作業開始直後は、理解した内容を短く要約してから実装に入ること。

### 推奨コマンド
- 開発開始時: `/start-dev`
- このプロジェクトの使い方確認: `/help-project`
- Claude Code 標準ヘルプ: `/help`

> 補足: `/` を入力すると、その環境で利用可能な組み込みコマンドとスキルの候補一覧を確認できます。

## トークン管理・セッション引き継ぎ

### コンテキスト超過の防止
- 会話が長くなり、コンテキストの残量が少なくなってきたと感じたら、**作業を中断する前に必ずハンドオフノートを作成してください。**
- ハンドオフノートの保存先: `docs/handoff-current.md`
- 作成後、ユーザーに「新しいセッションを開始して、`docs/handoff-current.md` を読んでから続きを始めてください」と伝えること。

### ハンドオフノートのフォーマット
```md
# ハンドオフノート - {日付}

## 現在の状態
- 何が完成しているか
- 何が未完了か

## 次にやること（優先順）
1. ...
2. ...

## 重要な注意事項・決定事項
- ...

## 関連ファイル
- ...
```

### TODO ファイルの管理
- 実装タスクは `TODO.md` に常に最新の状態で記録してください。
- 機能を実装したらすぐに `TODO.md` を更新すること。
- セッションの始めに `TODO.md` を確認して、どこから再開するかを把握すること。

## 作業後の自動整備ルール

**機能追加・バグ修正・設計変更を行ったら、ユーザーに言われなくても必ず以下を実施すること。**

### 1. ドキュメント更新
- `TODO.md`: 完了タスクに `[x]` を付け、新フェーズとして追記する
- `docs/handoff-current.md`: 現在の状態・次にやること・注意事項を最新化する
- `docs/spec.md`: 仕様に変化があった場合のみ更新する（軽微な変更は不要）

### 2. GitHub へのコミット＆プッシュ
- 作業がひと段落したら `git add` → `git commit` → `git push` まで行う
- コミットメッセージは `feat:` / `fix:` / `docs:` 等のプレフィックスを付けて日本語で書く
- 複数の小さな変更はまとめて1コミットにしてよい

### 3. タイミング
- ユーザーとの会話の区切り（質問への回答・機能実装の完了）ごとに実施する
- 実装途中の中断時は必ずハンドオフノートを更新してからコミットする

## 実装方針
- 既存コードのスタイル・構成に合わせること。
- 変更前に影響範囲を把握し、大きな変更は対象ファイルを明示してから進めること。
- 不明点があっても、まず `docs/spec.md` を優先的に参照すること。
- 仕様にない独自判断を入れる場合は、理由を短く説明すること。
- 可能ならテストまたは最低限の検証コマンドを実行すること。
